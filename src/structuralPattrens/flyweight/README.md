# 플라이웨이트 패턴 (Flyweight Pattern)

## 개요
플라이웨이트 패턴은 **여러 객체가 공통으로 쓰는 무거운 데이터를 공유**하여
**메모리 사용량을 줄이는** 구조(Structural) 패턴이다.

객체의 상태를 두 가지로 나누는 것이 핵심이다.
- **내재적 상태(intrinsic)**: 공유 가능한 불변 데이터 (몬스터 타입/메쉬/텍스처)
- **외재적 상태(extrinsic)**: 객체마다 달라지는 데이터 (좌표)

무거운 내재적 상태는 하나만 만들어 공유하고, 가벼운 외재적 상태만 객체마다 따로 갖는다.

## 왜 사용하는가
- 같은 종류의 객체를 대량 생성할 때, 공통 데이터를 매번 복제하면 메모리가 낭비된다.
- 공통 데이터를 **공유(캐싱)** 하면 몬스터 수가 늘어도 모델 객체는 타입 수만큼만 존재한다.
- 예제에서 몬스터 17마리(darkElf 10 + orc 7)를 만들어도 무거운 모델은 **2개**만 생성된다.

## 구성 요소 (역할)

| 역할 | 클래스 | 설명 |
|------|--------|------|
| **Flyweight** (공유 객체) | `MosterModel` | 공유되는 무거운 불변 데이터(내재적 상태). 반드시 불변이어야 함 |
| **FlyweightFactory** (공유 객체 관리자) | `MosterModelFactory` | 캐시(풀)에 공유 객체를 보관하고 재사용/생성을 관리 |
| **Context** (비공유 객체) | `Moster` | 객체마다 다른 데이터(외재적 상태, 좌표)를 갖고, 공유 모델을 참조 |
| **Client** (사용 예) | `Terrain`, `Main` | 팩토리에서 공유 모델을 받아 개별 데이터와 조합해 사용 |
| (보조) | `Memory` | 메모리 절약 효과 확인용 유틸 (패턴 구성 요소 아님) |

## 구조

```
      Client (Terrain/Main)
          │  getInstance("darkElf")
          ▼
   MosterModelFactory ──── cache(Map) : 타입별로 딱 1개씩 보관
          │  (없으면 생성, 있으면 재사용)
          ▼
      MosterModel (Flyweight) ◀────┐  ← 공유 (내재적 상태: 타입/메쉬/텍스처)
                                   │
      Moster (Context) ───────────┘  ← 개별 (외재적 상태: 좌표) + 모델 참조
```

## 핵심 포인트
- `MosterModel`은 공유되므로 **불변(final 클래스)** 으로 만들어 변경을 막는다.
  (하나를 바꾸면 이를 공유하는 모든 몬스터에 영향을 주기 때문)
- 클라이언트가 `new`로 직접 만들지 않고 **`MosterModelFactory`를 거치게** 하여 공유를 강제한다.
- `Moster`는 좌표(외재적 상태)만 개별로 갖고 무거운 모델은 참조만 하므로 가볍다.

## 실행 흐름 예시
```java
for (int i = 0; i < 10; i++) terrain.render("darkElf", x, y);
for (int i = 0; i < 7; i++)  terrain.render("orc", x, y);
Memory.print();
```
```
--- 몬스터 모델 객체 신규 생성 완료 ----   // darkElf 모델: 최초 1회만 생성
... (darkElf 몬스터 10개 생성, 모델은 재사용)
--- 몬스터 모델 객체 신규 생성 완료 ----   // orc 모델: 최초 1회만 생성
... (orc 몬스터 7개 생성, 모델은 재사용)
총 메모리 사용량 : ...MB
```
몬스터는 17마리지만 무거운 모델 객체는 타입별 1개씩(총 2개)만 생성되어 메모리를 절약한다.
