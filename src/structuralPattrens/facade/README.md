# 파사드 패턴 (Facade Pattern)

## 개요
파사드 패턴은 **복잡한 서브시스템을 하나의 통합된 인터페이스로 감싸서**,
클라이언트가 **단순한 메서드 호출만으로** 기능을 사용할 수 있게 하는 구조(Structural) 패턴이다.

`Facade`는 "건물의 정면"이라는 뜻으로, 내부의 복잡한 구조를 감추고 깔끔한 출입구만 보여준다.

## 왜 사용하는가
파사드 등장 전에는 클라이언트가 직접 이런 일을 해야 했다.
- 여러 서브시스템을 직접 호출
- 호출 순서를 지키고 중간 실패 처리도 직접 수행
- 시스템 간 의존 관계가 복잡해짐
- 내부 변경이 클라이언트 코드까지 영향

파사드는 이 복잡함을 하나의 창구 뒤로 감춘다.
- 클라이언트는 단순한 메서드 하나만 호출
- 내부 구조가 바뀌어도 외부 영향이 적음

## 구성 요소 (역할)

| 역할 | 설명 |
|------|------|
| **Facade** (통합 창구) | 서브시스템들을 감추고 단순한 인터페이스를 제공. 호출 순서·조율을 대신 처리 |
| **SubSystem** (서브시스템) | 실제 기능을 담당하는 하위 클래스들. 파사드를 통해서만 사용됨 |
| **Client** (사용 예) | 파사드의 단순한 메서드만 호출 |

## 구조

```
   Client ──▶ Facade ──┬──▶ SubSystem 1
                       ├──▶ SubSystem 2
                       └──▶ SubSystem 3
        (복잡한 호출·순서·검증을 Facade가 대신 처리)
```

---

## 예제 1 — v1 (기본 구조)

| 역할 | 클래스 |
|------|--------|
| **Facade** | `FacadeClass_v1` |
| **SubSystem** | `SubSystemClasses1_v1`, `SubSystemClasses2_v1`, `SubSystemClasses3_v1` |
| **Client** | `FacadeMain_v1` |

`facadeMethod()` 한 번의 호출로 세 서브시스템의 실행과 성공/실패 판단을 모두 처리한다.

## 예제 2 — v2 (사냥 예제)

| 역할 | 클래스 |
|------|--------|
| **Facade** | `HuntFacade` |
| **SubSystem** | `WeaponDamage`, `Attack` |
| **Client** | `FacadeMain_v2` |
| (보조) | `WeaponEnum` (무기 종류 열거형) |

클라이언트는 `executeAttack(weapon)`만 호출하면 되고,
파사드가 **데미지 계산 → 유효성 검사 → 공격**이라는 흐름을 내부에서 조율한다.

## 핵심 포인트
- 파사드는 서브시스템을 **없애는 것이 아니라 감싸서 단순화**한다. (직접 접근도 여전히 가능)
- 클라이언트는 서브시스템의 **호출 순서나 의존 관계를 몰라도 된다.**
- 내부 구조가 바뀌어도 파사드 인터페이스만 유지되면 클라이언트는 영향받지 않는다.
